# Copy Rules

Use these rules for all website copy, UI text, examples, and supporting language in this project.

The voice should be:
- direct
- helpful
- a bit cheeky

It should never be:
- vague
- self-important
- startup-generic
- cute at the expense of clarity

## Core Rule

Say what the skill does, what the visitor gets, or why it helps.

If a line does not do one of those three things, cut it.

## Tone Rules

1. Lead with the useful thing.
   Start with the action, output, or benefit.

2. Use normal words.
   Prefer plain English over “positioning” language.

3. Be specific.
   Name the topic, the output, or the result.

4. Keep the joke in the margins.
   A small wink is good. The main line still needs to work without the joke.

5. Sound like a smart person being clear, not a brand being clever.

6. Make the visitor feel helped, not marketed at.

## Hard Bans

Do not write lines like:
- “See the climb, not the clutter.”
- “A cleaner sequence.”
- “Built for moments like this.”
- “Unlock understanding.”
- “Make complex ideas simple.”

These sound polished but say almost nothing.

## Preferred Copy Pattern

Most lines should do one of these jobs:

- Explain the output
  Example: `The skill writes four versions of the same explanation: ages 4, 7, 12, and 16.`

- Explain the benefit
  Example: `People get the simple version first, then the more precise one.`

- Explain the use case
  Example: `Useful for docs, prompts, teaching, meetings, and anything that gets too technical too fast.`

- Explain the proof
  Example: `Here is what the skill outputs for “What is inflation?”`

## Writing Style

- Use short sentences.
- Prefer one idea per sentence.
- Prefer concrete nouns and verbs.
- Use contractions when they sound natural.
- Keep paragraphs to 1-3 lines.
- Cut repeated ideas across sections.

## Cheeky, Not Corny

Allowed:
- dry
- lightly self-aware
- office-adjacent
- one small wink in a section

Not allowed:
- trying too hard
- meme voice everywhere
- “quirky product” filler
- fake confidence

## Headlines

Headlines should be clear in one glance.

Good:
- `Explain complicated things in plain English.`
- `Four explanation levels. One idea.`
- `What the skill actually outputs.`

Bad:
- `Clarity that compounds.`
- `See the climb, not the clutter.`
- `Where complexity meets comprehension.`

## Supporting Copy

Supporting copy should answer the next obvious question.

Example:
- Headline: `What the skill actually outputs.`
- Support: `It rewrites one idea four times so a beginner can start small and keep going.`

Do not use supporting copy to restate the headline with fancier words.

## CTAs

CTAs should be blunt and useful.

Good:
- `Download the skill`
- `See example output`
- `Get the Markdown file`

Bad:
- `Start simplifying`
- `Experience clarity`
- `Explore the method`

## Examples Section Rules

The examples section must show output, not talk about output.

Always:
- use a real question or topic
- show the four age levels
- make each step meaningfully better
- keep labels obvious

Never:
- describe the framework instead of showing it
- explain the benefit in abstract language
- write copy that could apply to any product

## Editing Test

Before keeping any line, ask:

1. Would a first-time visitor learn something concrete from this?
2. Could this line belong to any SaaS homepage on the internet?
3. Is this trying to sound good instead of being useful?

If the answer to 2 or 3 is yes, rewrite it.

## Default Voice Reference

Aim for:
- teacher who respects your time
- helpful coworker with decent taste
- one eyebrow raised, not a comedy sketch

## Quick Rewrites

Weak: `The value is a cleaner sequence.`
Better: `The skill gives you four versions of the same explanation, starting simple and ending precise.`

Weak: `Complex ideas, made clear fast.`
Better: `Explain hard things in a way people actually understand.`

Weak: `One idea. Four passes. Zero wall of text.`
Better: `You get four versions of the same answer: 4, 7, 12, and 16.`
